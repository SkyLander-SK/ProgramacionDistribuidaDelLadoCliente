Escenario — Contexto del Problema

📢 Sofía Ramírez — Tech Lead, TechNova S.A. de C.V. "El equipo de proveedores de EcoMarket necesita urgente un monitor de disponibilidad de inventario. Tenemos una API que publica en tiempo casi-real el stock de cada producto en los almacenes. El problema: en este momento nadie sabe cuándo un producto cae por debajo del mínimo — nos enteramos tarde, los proveedores se quejan y perdemos ventas. Necesito que construyas el cliente que le da vida a este sistema: que consulte la API periódicamente, detecte cuándo hay cambios, y notifique al módulo de compras y al módulo de alertas de forma desacoplada. El servidor ya está listo — tu trabajo es construir el cliente que lo consume correctamente."

Tu tarea es implementar un cliente distribuido del lado del cliente para EcoMarket que monitoree la disponibilidad del inventario. El cliente debe consumir la API del servidor de inventario, usar programación asíncrona, implementar polling adaptativo con detección de cambios, y notificar a los módulos interesados mediante el patrón Observer — sin que ningún módulo dependa directamente de los detalles HTTP.

Códigos de respuesta posibles:

304 Not ModifiedEl inventario no cambió desde el último ETag enviado. No hay body. El cliente debe mantener el estado anterior.
400 Bad RequestHeader Authorization mal formado. No reintentar — hay un error en la petición.
401 UnauthorizedToken inválido o expirado. No reintentar — requiere acción del usuario.
503 Service UnavailableEl servidor está temporalmente sobrecargado. Aplicar backoff — reintentar después.
Timeout / ConexiónErrorError de red antes de recibir respuesta. El ciclo de polling debe continuar — registrar warning y esperar el próximo ciclo.
POST /alertas Registra una alerta de stock bajo en el sistema central

Observer 15%
2
Implementar _consultar_inventario() como función async
Función asíncrona que hace GET /inventario con los headers requeridos (Authorization + If-None-Match si ya hay un ETag guardado). Debe manejar cada código de respuesta de forma diferenciada: 200 actualiza el estado y el ETag; 304 no modifica nada; 4xx registra el error y no reintenta; 5xx incrementa el intervalo (backoff). Errores de red (Timeout, ConnectionError) deben capturarse: el ciclo de polling no debe detenerse por un fallo puntual de red.
HTTP 25% · Errores 25%
3
Implementar el ciclo de polling async con backoff adaptativo
Función asíncrona iniciar() con un ciclo while ejecutando que: (a) llama a _consultar_inventario(), (b) si hay datos nuevos (200 OK), compara con el estado anterior y llama a _notificar() si hubo cambios, (c) ajusta el intervalo — backoff si no hay cambios o hay 5xx; reset a base si hay cambios. El sleep debe ser no bloqueante (await asyncio.sleep() o equivalente). Implementa detener() que solo cambia una bandera para un cierre suave.
Async 20% · Polling 15%

Invariantes — Condiciones que deben cumplirse siempre

Estas reglas deben cumplirse independientemente del lenguaje o estructura que elijas. El evaluador verificará explícitamente cada una de ellas.

  • ● _consultar_inventario() nunca propaga una excepción al ciclo de polling — todos los errores se capturan dentro del método. El ciclo siempre continúa al siguiente ciclo, incluso tras un Timeout o ConnectionError.
  • ● Un 503 incrementa el intervalo (backoff). Un 400 o 401 registra el error pero no reintenta y no modifica el intervalo — si el header está mal formado, reintentar inmediatamente no tiene sentido.
  • ● El sleep del ciclo es no bloqueante: await asyncio.sleep(n) en Python, await new Promise(r => setTimeout(r, n)) en JS, o equivalente. time.sleep() bloqueante congela el event loop.
  • ● detener() solo cambia la bandera de control — no cancela la corrutina de forma forzada, no genera CancelledError ni excepciones en el ciclo.
  • ● _notificar() llama a actualizar(inventario) en todos los observadores de forma idéntica, sin condiciones if isinstance(obs, Tipo) ni switch por tipo. Si un observador lanza una excepción interna, el ciclo de notificación a los demás observadores no debe interrumpirse.
  • ● El body de la respuesta se valida antes de usarlo: si productos es null o el JSON no tiene el campo esperado, el cliente registra el problema y no llama a _notificar() con datos inválidos.

🚫 Condiciones de Fallo Crítico — Zero Tolerance

Crash silencioso que detiene el event loop: El cliente recibe un error de red y lanza una excepción no capturada que termina la corrutina principal. El ciclo muere sin registrar el error. → Criterio afectado: Manejo de errores + Programación asíncrona. Calificación máxima en esos rubros: Básico (50%).
Excepción no capturada en actualizar() de un observador que silencia al resto: Si el ciclo for obs in _observadores no está protegido por try/except, un observador con bug detiene la notificación de los siguientes. → Criterio afectado: Patrón Observer. Calificación máxima: Básico (50%).
time.sleep() bloqueante en el ciclo de polling: Congela el event loop durante el intervalo de espera; ninguna otra corrutina puede ejecutarse y el cliente no puede responder a detener(). → Criterio afectado: Programación asíncrona. Calificación máxima: Básico (50%).

Nota: un cierre controlado (ejecutando = False + salir del while limpiamente) es correcto y no activa estas condiciones. La distinción es fallo silencioso vs. cierre controlado.
